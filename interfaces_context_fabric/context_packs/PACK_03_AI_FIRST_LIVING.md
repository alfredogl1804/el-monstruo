## 3. Articulación del patrón

El usuario opera bajo un loop simbiótico:

```
1. Yo (humano) capturo contexto de mi vida en formatos legibles para la IA
   ↓
2. La IA absorbe el contexto, aprende patrones, reciproca con valor
   ↓
3. La IA me devuelve tiempo, claridad, decisiones mejor informadas
   ↓
4. Eso me libera para capturar más contexto, mejor calidad, más rápido
   ↓
5. Loop cerrado: el sistema se vuelve más útil cuanto más lo alimento
```

Esto NO es lo mismo que "AI-Native" (la IA está en todas partes). Es **AI-First Living**: el usuario reorganiza su forma de vivir para que la IA sea el centro gravitacional de su productividad cognitiva.

## 4. Transport Cero — el órgano que materializa AI-First Living

Transport Cero es el motor de ingesta AI-first que precede a Telegram, Flutter, Command Center, La Forja y WhatsApp. **No es ingesta de archivos.** Es ingesta con **juicio ontológico** — 8 preguntas que cada input atraviesa antes de aterrizar en una superficie.

### 4.1. Las 8 preguntas obligatorias del Transport Cero

| # | Componente | Pregunta de juicio |
|---|---|---|
| 1 | Ontological Intake | ¿Qué tipo de cosa es esto? |
| 2 | Capture Heat Check | ¿Cuán urgente / prioritario es el momento de captura? |
| 3 | Reconstruction Sufficiency Score (0-5) | ¿La metadata permite reconstruir el contexto en N años? |
| 4 | Microcontext Prompt | ¿Cuál es el contexto mínimo asociado (lugar, gente, estado mental)? |
| 5 | Rhythm Gate | ¿Debe interrumpir el ritmo soberano de Alfredo? |
| 6 | Delegation Router | ¿Quién/qué debe ejecutar lo que esto implica? |
| 7 | Focus Guard | ¿Está alineado con los 5 objetivos vivos? |
| 8 | Memory Candidate | ¿Debe recordarse, expirar, no indexarse o pasar a revisión? |

Las preguntas 5, 6, 7 derivan directamente de los 5 órganos latentes detectados en la libreta pre-IA (PACK_11 sección 8). La pregunta 3 deriva del principio PRE-IA-006 (Captura Reconstruible). La pregunta 8 deriva de PRE-IA-008 (Memoria de Estado Mental).

### 4.2. Reconstruction Sufficiency Score — escala 0 a 5

El RSS mide cuánto contexto puede reconstruirse desde una captura años después. Pre-IA, la libreta de Alfredo intentaba esto manualmente con notas tipo "estado del día". Ahora se formaliza:

| Nivel | Definición | Ejemplo |
|---|---|---|
| **0** | Solo el bruto (texto/imagen/audio) sin metadata. | Foto sin EXIF |
| **1** | Bruto + timestamp. | Nota con fecha |
| **2** | Bruto + timestamp + ubicación. | Foto con GPS |
| **3** | Bruto + timestamp + ubicación + entidad mencionada (personas, proyecto, objeto). | Audio que identifica hablantes |
| **4** | Nivel 3 + estado mental/clima emocional capturado. | Captura ambient con HRV correlation |
| **5** | Nivel 4 + propósito declarado de la captura ("estoy guardando esto porque..."). | Reflexión Cronos con why explícito |

**Política operativa propuesta:** Transport Cero rechaza capturas RSS<2 salvo casos triviales (screenshots de chat). RSS≥3 es objetivo para todo input que vaya al río de Cronos. RSS=5 es exigido para Modo Cripta (legado).

### 4.3. Flujo de Transport Cero

```
input crudo (voz / foto / texto / archivo / pantalla / sensor)
  ↓
ontological intake (qué tipo) ............................ pregunta 1
  ↓
capture heat check (urgencia del momento) ................ pregunta 2
  ↓
reconstruction sufficiency score (0-5) ................... pregunta 3
  ↓ si RSS<2 → solicitar enriquecimiento o descartar
microcontext prompt (lugar, gente, estado mental) ........ pregunta 4
  ↓
rhythm gate (¿interrumpe?) ............................... pregunta 5
  ↓ si interrumpe → escala a transport visible
delegation router (¿quién hace qué?) ..................... pregunta 6
  ↓
focus guard (¿alineado con 5 objetivos?) ................. pregunta 7
  ↓ si no alineado → backlog o ignorar
memory candidate (¿persiste?) ............................ pregunta 8
  ↓ destino:
       - Cronos (capa correspondiente)
       - Memento (operativo del agente)
       - Catastro
       - Sin indexar
       - Revisión humana
       - Expirar TTL
```

## 5. Implicaciones para interfaces (especulativas, requieren validación T1)

### 5.1. La interfaz no es para el humano — es para la IA

Los formatos en los que el humano captura información (notas, fotos, voz, gestos) deben ser optimizados para que la IA los **digiera**, no para que el humano los **revise después**. Esto invierte el design pattern habitual.

### 5.2. El éxito se mide en "fricción de captura", no en "engagement"

Cuanto más rápido el usuario puede dejar un dato en el Monstruo y **olvidarse** de él, mejor. Cero retención del usuario en pantalla — todo lo opuesto a la economía atencional actual.

### 5.3. La interfaz tiene que **anticipar el formato que la IA necesita**

Si la IA necesita una transcripción de voz con timestamps + ubicación + identificación de hablantes para hacer su trabajo bien, la interfaz captura todo eso **automáticamente, sin pedirle nada al humano**. El humano solo habla.

### 5.4. Cero mejor que poco

Si el usuario tiene que decidir qué formato usar para capturar algo, la interfaz ya falló. La IA decide el formato, el usuario solo provee el bruto. Esto es PRE-IA-007 (Fricción Mínima) cumplido.
