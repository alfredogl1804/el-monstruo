## TL;DR — el dato duro

**92 tablas en producción tienen Row Level Security desactivado.** Incluye:

- **`tool_secrets` (8 filas) — P0 CRÍTICO.** Nombre sugiere que contiene credenciales/keys. Sin RLS, cualquiera con la `anon key` (que está embebida en el frontend) puede leerlas.
- **`user_dossier` (1 fila) — P1 ALTA.** Datos personales del owner.
- **5 tablas P1 con datos sensibles del embrión:** `embrion_budget_state`, `episodic_memory`, `error_memory`, `error_memory_patterns`, `monstruo_memory`, `frontend_sessions`.
- 85 tablas P2 con datos operativos: checkpoints (49,734 + 21,761 filas), chunks (25,892 filas), articulos (39,065 filas), menciones_bm (20,514 filas), scheduled_tasks (15,191 filas), etc.

`embrion_memoria` SÍ tiene RLS habilitado — única tabla del stack del embrión protegida.

## Riesgo concreto

La `anon key` de Supabase es **pública por diseño** — se embebe en el frontend (Flutter, web, etc.) para que cualquier cliente conecte. Sin RLS, esa key vale como key de admin. Cualquiera que la extraiga del bundle puede:

- `SELECT * FROM tool_secrets` → robar las 8 credenciales
- `INSERT/UPDATE/DELETE` en cualquiera de las 92 tablas
- Leer la memoria entera del Monstruo (`monstruo_memory`, 68 filas, 168 KB)
- Leer el dossier personal del owner

Esto NO es problema de Cowork ni de Manus. Es deuda heredada del setup inicial de Supabase. **No se puso RLS desde el día 1.**

## Acción propuesta — Sprint S-002.5 (mini-sprint hardening RLS)

Independiente del sprint EMBRION-NEEDS-001 que está corriendo. Tres tareas:

### Tarea 1 — P0 Inmediato (1-2 horas)

```sql
-- Activar RLS en tablas críticas
ALTER TABLE public.tool_secrets ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.user_dossier ENABLE ROW LEVEL SECURITY;

-- Política mínima: solo service_role puede leer/escribir
CREATE POLICY "service_role_only" ON public.tool_secrets
  FOR ALL USING (auth.role() = 'service_role')
  WITH CHECK (auth.role() = 'service_role');

CREATE POLICY "service_role_only" ON public.user_dossier
  FOR ALL USING (auth.role() = 'service_role')
  WITH CHECK (auth.role() = 'service_role');
```

Esto bloquea inmediatamente el acceso vía `anon key`. Solo el kernel (que usa `service_role`) sigue accediendo. **Cero impacto operativo, máxima protección.**

### Tarea 2 — P1 stack del embrión (2-4 horas)

Mismo patrón para las 5 tablas P1 del embrión + `monstruo_memory`. Validar que el kernel sigue funcionando con `service_role` (debería).

### Tarea 3 — P2 sistemático (1-2 días)

Auditar las 85 tablas P2 categorizándolas: cuáles necesitan acceso anon (frontend), cuáles solo service_role, cuáles roles intermedios. Aplicar RLS por grupo.
