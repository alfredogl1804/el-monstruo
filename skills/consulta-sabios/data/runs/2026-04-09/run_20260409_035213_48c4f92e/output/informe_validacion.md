### 🆕 EU AI Act obligations for agentic systems and marketplaces

# Obligaciones del EU AI Act para sistemas agenticos y plataformas

El **EU AI Act entra en vigor completamente el 2 de agosto de 2026**, tan solo 4 meses desde hoy[1][2]. Para sistemas agenticos, las obligaciones varían según su nivel de riesgo.

## Clasificación de agentes

Los sistemas agenticos se dividen en 4 niveles de riesgo[1]. Los agentes de **alto riesgo** (como selección de personal o CV screening) requieren las obligaciones más extensas: sistema de gestión de riesgos, gobernanza de datos, documentación técnica detallada, registro automático de actividad, transparencia, supervisión humana y cumplimiento de ciberseguridad[1].

En contraste, agentes de **riesgo limitado** (chatbots de atención al cliente, sistemas de reservas) solo deben incluir avisos de transparencia indicando que son IA[1].

## Obligaciones clave que entran en vigor

Antes de agosto de 2026, los proveedores y operadores de sistemas de alto riesgo deben[2][3]:

- **Marcado CE**: obtener conformidad europea
- **Registro en base de datos UE**: registrar públicamente el sistema antes de lanzarlo
- **Evaluación de conformidad**: documentar cumplimiento normativo
- **Logs de actividad**: conservar registros mínimo 6 meses[3]
- **Supervisión humana**: garantizar control humano sobre decisiones

## Implicaciones para plataformas

El Reglamento "regula la inteligencia artificial allí donde se usa y donde genera efectos"[5]. Las plataformas que ofrecen agentes deben revisar contratos con proveedores de IA, establecer mecanismos de notificación de incidentes y responsabilidades compartidas[5].

## Multas

Incumplimiento: hasta **35 millones de euros o 7% de facturación global**[6].

**Acción recomendada**: Las empresas españolas deben clasificar sus agentes inmediatamente y completar evaluaciones de conformidad antes de agosto de 2026[3].

### 🆕 GitHub code search and rate limits for large-scale SKILL.md discovery

**GitHub Code Search Rate Limits for Large-Scale Discovery**

Para descubrimientos a gran escala como búsquedas de archivos SKILL.md, los límites de tasa de GitHub representan una restricción significativa. La API de búsqueda de código tiene un límite de **10 solicitudes por minuto**[1], separado de otros tipos de búsqueda.

**Limitaciones Prácticas**

Cada solicitud a la API de búsqueda de código retorna aproximadamente 100 resultados por página[2]. Por lo tanto, recuperar 1,000 resultados requiere 10 solicitudes, agotando completamente el límite de un minuto[2]. Un usuario reportó este problema específicamente: al intentar obtener 1,000 resultados de código con `gh search code --limit 1000`, recibió un error HTTP 403 después de solo 10 páginas de resultados[2].

**Requisitos Actuales (2026)**

Desde marzo de 2026, todos los endpoints de la API de búsqueda de código requieren autenticación[1]. Esto afecta particularmente a consultas con alcance de repositorio. El ordenamiento de resultados de búsqueda también fue deprecado; ahora todos los resultados se ordenan por relevancia automáticamente[1].

**Estrategias Recomendadas**

Para búsquedas a gran escala, GitHub recomienda[1]:
- Utilizar webhooks para monitorear cambios
- Implementar GitHub Advanced Security en lugar de consultas repetitivas
- Paginar eficientemente respetando el límite de 10 solicitudes/minuto
- Considerar implementar backoff exponencial cuando se alcance el límite

**Conclusión**

El límite de 10 solicitudes por minuto es restrictivo para descubrimientos masivos. Un inventario completo de archivos SKILL.md en múltiples repositorios requeriría optimización cuidadosa, posiblemente dividiendo búsquedas en consultas más específicas o utilizando alternativas como clonación de repositorios locales para análisis sin restricciones de API[1][2].

### 🆕 License compatibility of imported skills for commercial redistribution

Los resultados de búsqueda disponibles no contienen información específica sobre compatibilidad de licencias de "habilidades importadas" para redistribución comercial. Sin embargo, puedo aplicar los principios generales de compatibilidad de licencias que sí están documentados.

## Principios aplicables

La **compatibilidad de licencias** se refiere a la posibilidad legal de combinar código o trabajos bajo diferentes licencias.[4][5] Los requisitos contradictorios entre licencias pueden hacer imposible legalmente combinar trabajos para crear y publicar un nuevo programa.[4]

### Categorías de licencias relevantes:

Las licencias se dividen en tres clases:[2]

- **Licencias permisivas** (como MIT o Apache 2.0): son compatibles con prácticamente todo, incluyendo licencias propietarias. Permiten combinar código bajo licencias diferentes sin restricciones significativas.[2][5]

- **Licencias copyleft**: requieren que los trabajos derivados mantengan la misma licencia, lo que generalmente las hace incompatibles con licencias propietarias.[2][4]

- **Licencias intermedias**: ponen condiciones en agregar código propietario pero no lo prohíben completamente.[2]

### Para redistribución comercial:

Las licencias con cláusulas **NonCommercial** presentan restricciones específicas: no pueden combinarse con licencias que permitan uso comercial, ya que esto crearía conflictos irreconciliables.[1][3] Del mismo modo, las licencias **NoDerivatives** impiden que los trabajos derivados se distribuyan públicamente.[3]

Según la FSF, **no existe compatibilidad simétrica**: una licencia puede permitir relicenciamiento a otra sin que lo inverso sea cierto.[2]

## Recomendación

Para redistribuir comercialmente trabajos importados, debe verificarse explícitamente que todas las licencias involucradas sean compatibles con: a) combinación o derivación, y b) uso comercial. Las licencias permisivas ofrecen mayor flexibilidad para estos escenarios.[5]

### 🆕 Data transfer and data residency implications when using external agent tools

El uso de herramientas de agentes externos genera riesgos significativos de transferencia internacional de datos y exposición de información personal, especialmente cuando estos agentes integran servicios fuera del Espacio Económico Europeo.

## Riesgos de transferencia de datos con agentes IA

Cuando los agentes consultan servicios externos o acceden a información interna, **los datos personales pueden salir del territorio de la UE hacia repositorios no previstos**[5]. La AEPD señala que la integración de servicios internos, modelos de lenguaje y herramientas externas genera **problemas de interoperabilidad, cambios inesperados en términos de servicio y dificultades para garantizar continuidad normativa**[5].

## Marco normativo aplicable

Según el RGPD, **los datos personales solo pueden transferirse fuera de la UE si se garantiza un nivel de protección equivalente**[1]. Las transferencias internacionales requieren mecanismos legales como decisiones de adecuación (artículo 45) o cláusulas contractuales tipo (artículo 46)[1]. Tras la sentencia Schrems II, estas cláusulas exigen análisis adicionales del nivel de protección efectivo en el país destino[1].

## Medidas de control contractual

Los Acuerdos de Procesamiento de Datos (DPA) deben **prohibir explícitamente a proveedores usar datos para entrenar modelos públicos** y exigir registros de auditoría granulares[4]. Los responsables deben exigir que proveedores hereden controles de acceso empresariales en lugar de mantener listas de permisos separadas[4].

## Documentación y evaluación de riesgos

Las organizaciones deben evaluar el riesgo de acceso por autoridades extranjeras, implementar medidas técnicas y organizativas adicionales, documentar análisis realizados e incorporar transferencias al registro de actividades[1]. Una **evaluación de impacto relativa a protección de datos** es necesaria, especialmente con categorías especiales de datos[1].

A mayor número de servicios integrados, mayor es la **complejidad y el riesgo de exposición** en repositorios no controlados[5].
